2026-02-20

Major changes
- Match flow is now centralized in `main/match_director.gd` with explicit states:
  `LOBBY`, `RUNNING`, `WON`, `LOST`, `RESETTING`.
- MatchDirector now acts as the single server-side source of truth for:
  score, lives, deaths, and shared objective progress.
- Added/solidified event-style API on MatchDirector:
  `register_player_spawn`, `report_player_fell`, `report_enemy_killed`,
  `report_objective_progress`, `request_respawn`, `set_player_lives`.
- MatchDirector respawn permission is now enforced centrally via `request_respawn`.
- Match snapshot broadcasting remains centralized through one RPC
  (`_receive_snapshot`) consumed by UI.

Gameplay script refactor
- `main/fall_checker.gd` now reports falls to MatchDirector instead of owning
  its own independent life counters.
- `enemies/bee_bot.gd` now reports kills through
  `report_enemy_killed("bee_bot", killer_id)` instead of direct score mutation.
- `player/player.gd` now synchronizes enemy-hit life changes back into
  MatchDirector to keep counters coherent.
- `ui/ui.gd` now reads connection info + MatchDirector snapshot only
  (removed separate FallChecker lives feed).

Tests
- Extended `test/test_match_director.gd` with life-loss and enemy-kill objective tests.
- Updated `test/test_fall_lives.gd` to validate FallChecker + MatchDirector integration.
- Full GUT suite currently passes in headless run (20/20).

Bugfix update
- Fixed a life-counter rebound where bomb damage could reduce player lives by 1 and
  then immediately restore it due to stale server-side lives state.
- Root cause: damage was applied on node authority peer, while MatchDirector lives
  remained unchanged on server; FallChecker later re-synced stale lives back.
- `player/player.gd` enemy damage flow is now server-authoritative and updates both:
  1) MatchDirector (`set_player_lives`) and
  2) owning player peer (`set_lives` sync).
- Extended integration coverage in
  `test/integration/test_bomb_player_integration.gd` to assert:
  - bomb hit decrements player lives,
  - director lives are updated identically,
  - FallChecker sync does not cause lives to bounce back.

Gameplay update
- Added slope sliding behavior in `player/player.gd`:
  - `slide_start_angle_deg`: slope angle where sliding starts.
  - `slide_block_angle_deg`: slope angle where uphill input is fully blocked.
  - `slope_slide_accel`: downhill tangential acceleration on steep slopes.
  - `slope_slide_friction`: damping to keep slide speed stable.
- Implementation details:
  - Uphill input is progressively reduced between start/block angles.
  - A downhill slide impulse is applied when standing on steep ground.
  - Flat terrain and gentle slopes keep normal controls.

Sliding visual feedback update
- Added explicit sliding visual state for players:
  - synchronized flag `_is_sliding_sync` replicated via `player/player.tscn`.
  - authority-driven `set_sliding(...)` in `player/model/character_skin.gd`.
- While sliding:
  - locomotion animation is paused from walk/run state updates,
  - fallback pose uses existing `fall` animation state for readability,
  - `CharacterRotationRoot` gets a configurable forward tilt for clearer feedback.
- New player tuning parameters:
  - `slide_visual_tilt_deg`
  - `slide_visual_lerp_speed`

Sliding behavior tuning update
- Removed static `fall` pose fallback during slope sliding; sliding now keeps
  locomotion animation active for better readability.
- Downhill motion is now speed-target based:
  - accelerates toward `slope_downhill_speed` only along downhill direction,
  - does not boost uphill movement,
  - damps lateral drift only (`slope_lateral_damping`).
- Sliding visual state is now tied to actual downhill speed
  (`slide_visual_speed_threshold`), so stopping on a slope returns to normal
  standing/idle presentation.

Slope contact stability update
- Improved player-to-ground contact on steep slopes in `player/player.gd`:
  - added `slope_floor_snap_length` (default `0.55`),
  - applies `apply_floor_snap()` while descending and not jumping.
- This reduces the "hovering above slope" effect and limits contact loss
  during steep downhill movement.

Downed / revive loop update
- Added a server-authoritative downed revive mechanic using `Coin`.
- `Player` changes:
  - dead players now remain visible ("ghost view") instead of being hidden,
  - dead players are added to `downed_players` group,
  - added `can_be_revived()` and `try_revive_with_coin()` methods,
  - revive restores exactly `1` life (not full lives) and re-enables control.
- `Coin` changes:
  - server scans nearby downed players and can auto-target them,
  - if any downed player exists, normal alive pickup is blocked to preserve revive usage,
  - on collect, coin first attempts revive; otherwise falls back to normal pickup.
- `FallChecker` changes:
  - when a fall causes life to reach `0`, player is respawned back on the map as downed
	so teammates can revive (instead of staying unreachable under the map).
- Added integration coverage:
  - `test/integration/test_coin_revive.gd`
  - updated `test/test_fall_lives.gd` for downed-respawn-at-zero behavior.

Downed/revive polish update
- Coin consumption is now synchronized across peers:
  - coin becomes unavailable/hidden immediately when consumed,
  - coin is not reusable after a revive.
- Coin consume sound (`player/coin/audio/completetask_0.mp3`) is now triggered
  at actual revive/consume time for all peers.
- Added clearer downed visual feedback on player nickname labels:
  - suffix `"[DOWNED]"`,
  - red-tinted nickname color while downed.

Cooperative pull-cube puzzle update
- Added a new pull interaction on green rigid cubes (`main/rigid_body_3d.gd`):
  - left click from a player now toggles attach/detach on a pullable cube
	(implemented in `player/player.gd` via camera raycast),
  - each attached player contributes a traction vector from cube to player,
  - vectors naturally cancel when players pull in opposite directions,
  - vectors accumulate when players pull from the same side.
- Added visual state feedback synchronized to all peers:
  - `IDLE` (green),
  - `ATTACHED` (yellow),
  - `COOP` (orange),
  - `GOAL` (blue, cube frozen in reactor zone).
- Added reactor objective check:
  - cubes auto-detect the scene node named `reactor`,
  - when inside `reactor_goal_radius`, cube marks goal reached and freezes.
- Replication update in `main/main.tscn`:
  - `SceneReplicationConfig_fwcvl` now also replicates `._pull_state_sync`.
- Added test coverage in `test/test_pull_cube.gd`:
  - pull vectors accumulate on same-side pulls,
  - pull vectors cancel on opposite pulls,
  - reactor goal state transitions to `GOAL` correctly.
- Added a visible `RayCast3D` (`PlayerCamera/PullRay`) that mirrors `_try_toggle_pull_cube()`
  so the left-click ray appears in Debug mode; the ray is updated every call based
  on `pull_interaction_distance`.
- When two players stay aligned and the cube is in `COOP` (orange) state, it now
  auto-drives toward the `reactor` once their pull direction aligns; while auto-moving
  the emission energy grows (`auto_move_glow_boost`) so the team can see the assist
  engage before the cube reaches `GOAL`.
- Door/bomb synchronization fix:
  - `main/bomb_door.gd` now marks `_set_open_state` as an `all_peers` RPC so the
	server-driven opening state is propagated to every client instead of staying
	local to the authority.
 

Player architecture refactor (domain components)
- Refactored `player/player.gd` into an orchestrator (214 lines) and extracted
  gameplay domains into dedicated components under `player/components/`:
  - `player_movement.gd` (movement, slope rules, animation driving, push physics),
  - `player_combat.gd` (melee attack + server-authoritative enemy hit handling),
  - `player_lifecycle.gd` (lives, dead/downed state, revive flow),
  - `player_net_sync.gd` (replicated sync properties + client interpolation),
  - `player_interactions.gd` (bomb spawn/throw, pull-cube interaction, coin collect).
- Preserved public `Player` API and RPC entrypoints used by existing systems/tests:
  - `spawn_bomb`, `respawn`, `set_dead_state`, `set_lives`, `_collect_coin`,
	`_request_enemy_hit_on_server`, `damage`, `try_revive_with_coin`, etc.
- Goal: improve readability, reduce single-file cognitive load, and make future
  changes safer by isolating movement/combat/network/lifecycle concerns.
- Validation: full GUT test suite passes after refactor (`22/22`).
