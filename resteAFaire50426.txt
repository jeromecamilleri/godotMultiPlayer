  Ce qui est globalement sain

  - L’architecture reste cohérente avec ARCHITECTURE_PROJET.md et AGENTS.md :
	  - Player reste une façade
	  - les composants portent la logique
	  - MatchDirector reste l’état global
	  - beaucoup d’objets persistants ont maintenant un vrai traitement late join
  - La couverture de test est devenue bonne sur les flux sensibles :
	  - inventaire/coffre
	  - bombe/mur
	  - mission cube
	  - stress réplication
	  - scarabées
  - Les tests UI E2E sous Xvfb sont maintenant une vraie force du projet.

  Les zones que je renforcerais en priorité

  - Stabilité réseau des objets persistants
	  - faire un audit final de tous les interactifs restants
	  - surtout tout ce qui change d’état sans disparaître complètement
	  - objectif : chaque objet persistant doit suivre explicitement la checklist late join
  - Homogénéité ennemis
	  - aujourd’hui abeilles et scarabées suivent des logiques proches mais pas totalement unifiées
	  - je créerais une base commune d’ennemi autoritaire ou au moins une convention partagée :
		  - activation/désactivation
		  - damage signature
		  - resync late join
		  - score report
		  - director integration
  - Nettoyage de la scène main
	  - main/main.tscn concentre beaucoup trop d’éléments gameplay
	  - plusieurs objets critiques sont encore repérés par nom ou structure de scène
	  - je déplacerais progressivement les setups gameplay dans des sous-scènes dédiées :
		  - bloc mission cube
		  - bloc interactifs
		  - bloc ennemis
  - Conventions de groupes / IDs stables
	  - tu as eu raison plus tôt sur ce sujet
	  - il faut continuer :
		  - groupes stables pour objets gameplay
		  - test_id stables pour UI
	  - moins de dépendance aux NodePath fragiles
  - Performance réseau
	  - le coffre a déjà reçu un premier niveau d’optimisation
	  - l’étape suivante évidente reste :
		  - deltas de slots modifiés
		  - moins de snapshots complets
		  - moins de RPC fiables redondantes
  - Observabilité
	  - le projet devient assez riche pour justifier un vrai mode debug réseau/gameplay
	  - exemple :
		  - cible des ennemis
		  - état du directeur de partie
		  - révisions d’inventaire
		  - événements récents de sync
	  - ça accélérerait énormément le debug multi.

  Les risques que je vois encore

  - Certaines logiques restent un peu trop couplées à la scène réelle.
  - Le serveur sans vrai joueur local rend certains raisonnements gameplay/test un peu ambigus.
  - Les scénarios E2E sont puissants mais encore coûteux à maintenir si la map change souvent.
  - Le projet commence à accumuler plusieurs “modes test” via variables d’environnement ; il faudra éviter qu’ils deviennent
	une seconde architecture cachée.

  Ce que je ferais ensuite, dans l’ordre

  1. Ajouter un mini “debug overlay” réseau/gameplay activable.
  2. Formaliser des groupes gameplay stables dans main.
  3. Unifier davantage les conventions enemy director + enemy instance.
  4. Continuer l’optimisation réseau du coffre en deltas.
  5. Découper main/main.tscn en sous-scènes de mission plus modulaires.

  - la base est maintenant bonne
  - le plus gros gain à venir n’est pas une nouvelle feature isolée
  - c’est la robustesse structurelle : moins de fragilité de scène, plus d’observabilité, et une logique ennemis/réplication
	encore plus homogène
